# AGENTS.md

## Communication

默认使用中文回复用户。保持回复简洁、直接、可操作；除非用户要求详细说明，否则避免冗长解释。

## 1. 思考先行

**不要假设，不要隐藏困惑，暴露权衡取舍。**

实施前：
- 明确陈述你的假设。不确定就问。
- 如果存在多种理解，展示出来——不要默默选择一种。
- 如果存在更简单的方案，说出来。必要时提出反对意见。
- 如果有不清楚的地方，停下来。指出困惑点。提问。

## 2. 简洁优先

**最少的代码解决问题，不做推测性开发。**

- 不做超出要求的功能。
- 单次使用的代码不做抽象。
- 不做未被请求的"灵活性"或"可配置性"。
- 不为不可能的场景添加错误处理。
- 如果 200 行能用 50 行搞定，就重写。

检验标准："一个高级工程师会说这太复杂了吗？"如果是，就简化。

## 3. 精准修改

**只改必须改的，只清理自己造成的问题。**

编辑现有代码时：
- 不要"改进"相邻的代码、注释或格式。
- 不要重构没问题的代码。
- 匹配现有风格，即使你会用不同方式。
- 如果发现无关的死代码，提一下——不要删它。

当你的改动产生了孤立代码时：
- 删除你的改动导致不再使用的 import/变量/函数。
- 不要删除已有的死代码，除非被要求。

检验标准：每一行改动都应直接追溯到用户的请求。

## 4. 目标驱动

**定义成功标准，循环直到验证通过。**

将任务转化为可验证的目标：
- "添加验证" → "为无效输入编写测试，然后让它们通过"
- "修复 bug" → "编写复现测试，然后让它通过"
- "重构 X" → "确保重构前后测试都通过"

多步骤任务需陈述简要计划：
```
1. [步骤] → 验证: [检查点]
2. [步骤] → 验证: [检查点]
3. [步骤] → 验证: [检查点]
```

## Git 提交规范

- 提交信息必须使用中文编写
- 遵循 Conventional Commits 标准：`<type>[可选作用域][!]: <描述>`，type 包括 `feat`、`fix`、`docs`、`style`、`refactor`、`perf`、`test`、`build`、`ci`、`chore`、`revert`
- 提交正文需包含详细变更说明，解释改动前的主要问题以及改动如何解决它
- 按功能点拆分提交；每个提交应包含一个独立的功能、修复、文档/规则变更或其他 Conventional Commits 类型
- 完成任务并确认无问题后，自动提交相关代码变更，无需等待额外的提交请求
